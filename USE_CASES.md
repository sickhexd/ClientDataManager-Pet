<table>
    <tr>
        <th>Название</th>
        <td><b>UC-1. Регистрация клиента и создание счёта</b></td>
    </tr>
    <tr>
        <th>Описание</th>
        <td>Пользователь заполняет данные в форме регистрации, система сохраняет нового пользователя, а затем открывает счёт для клиента в банке. Все данные передаются по защищенному каналу (например, через HTTPS), а пароли хешируются для безопасности.</td>
    </tr>
    <tr>
        <th>Акторы</th>
        <td>Пользователь, Система</td>
    </tr>
    <tr>
        <th>Начальное состояние</th>
        <td>Пользователь находится на странице регистрации. У него нет учётной записи в системе и нет открытого счёта.</td>
    </tr>
    <tr>
        <th>Триггер</th>
        <td>Пользователь решил зарегистрироваться в системе (нажал "Зарегистрироваться").</td>
    </tr>
    <tr>
        <th>Успешный сценарий</th>
        <td>
            1. Пользователь заполняет форму регистрации (ФИО, email, мобильный телефон, паспортные данные).<br>
            2. Система проверяет корректность введённых данных (формат email, номера телефона, паспортных данных).<br>
            3. Система сохраняет нового пользователя в базе данных.<br>
            4. Система автоматически открывает новый счёт для пользователя в банке.<br>
            5. Система отправляет сообщение на email и/или номер телефона о успешной регистрации и открытии счёта.<br>
            6. Все изменения данных сохраняются в журнале аудита с указанием времени и ответственного сотрудника.
        </td>
    </tr>
    <tr>
        <th>Альтернативный сценарий</th>
        <td>
            (Альт.1: Email или телефон уже зарегистрированы, ветвление от шага 2)<br>
            2а. Система обнаруживает, что указанный email или телефон уже зарегистрированы.<br>
            2b. Система отображает сообщение "Пользователь с таким email/телефоном уже существует".<br>
            2с. Сценарий завершён неуспешно: регистрация и открытие счёта не выполнены.
        </td>
    </tr>
    <tr>
        <th>Негативный сценарий</th>
        <td>
            (Негативный: Ошибка сети при отправке формы)<br>
            1а. Система не получила данные формы из-за сбоя сети.<br>
            1b. Система выводит пользователю сообщение "Ошибка регистрации. Повторите попытку позже."<br>
            1с. Сценарий прерван: пользователь не зарегистрирован, счёт не открыт.
        </td>
    </tr>
    <tr>
        <th>Конечное состояние</th>
        <td>
            При успешном сценарии — новый пользователь создан в системе, счёт открыт, и пользователю отправлено подтверждение.<br>
            При альтернативном/негативном — новый пользователь не создан, счёт не открыт (система без изменений).
        </td>
    </tr>
    <tr>
        <th>Критерии приемки</th>
        <td>
            Пользователь с уникальными email и номером телефона может успешно зарегистрироваться, его данные сохраняются, и счёт открывается. Система проверяет валидность данных; при ошибках показывает понятные подсказки. Пользователь получает сообщение на email и/или номер телефона о успешной регистрации и открытии счёта.<br>
            Все изменения данных клиента фиксируются в журнале аудита.
        </td>
    </tr>
    <tr>
        <th>Связанные требования</th>
        <td>
            BR-1.1 "Возможность добавления новых клиентов с указанием персональных данных: ФИО, дата рождения, адрес, контактная информация, паспортные данные."<br>
            BR-1.2 "Валидация данных на этапе регистрации (например, проверка на корректность адреса электронной почты, номера телефона)"<br>
            BR-2.1 "Создание счёта для нового пользователя."<br>
            BR-4.1 "Все данные должны передаваться по защищённому каналу (например, через HTTPS)."<br>
            BR-5.1 "Все изменения должны быть зафиксированы в журнале аудита."
        </td>
    </tr>
</table>


<table>
    <tr>
        <th>Название</th>
        <td><b>UC-2. Редактирование данных клиента.</b></td>
    </tr>
    <tr>
        <th>Описание</th>
        <td>Ответственный менеджер или сам пользователь заполняет данные в форме для изменения, система сохраняет обновленные данные, и пользователю отправляется уведомление о успешном изменении данных. Все изменения должны быть зафиксированы в журнале аудита.</td>
    </tr>
    <tr>
        <th>Акторы</th>
        <td>Ответственный менеджер, Пользователь, Система</td>
    </tr>
    <tr>
        <th>Начальное состояние</th>
        <td>Пользователь или менеджер находятся на странице для обновления данных. Пользователь является клиентом банка и имеет открытый счёт.</td>
    </tr>
    <tr>
        <th>Триггер</th>
        <td>Пользователь или менеджер решили обновить данные в банке (нажали "Обновить данные").</td>
    </tr>
    <tr>
        <th>Успешный сценарий</th>
        <td>
            1. Пользователь или менеджер заполняют форму с данными, которые необходимо изменить (ФИО, email, мобильный телефон, паспортные данные).<br>
            2. Система проверяет корректность введённых данных (формат email, номера телефона, паспортных данных).<br>
            3. Система обновляет необходимые данные в базе данных при успешной валидации.<br>
            4. Система отправляет пользователю сообщение о успешном изменении данных на почту и/или на телефон.<br>
            5. Все изменения данных клиента сохраняются в журнале аудита.
        </td>
    </tr>
    <tr>
        <th>Альтернативный сценарий</th>
        <td>
            (Альт.1: Email, телефон или паспортные данные уже зарегистрированы, ветвление от шага 2)<br>
            2а. Система обнаруживает, что указанный email, телефон или паспортные данные уже зарегистрированы.<br>
            2b. Система отображает сообщение "Пользователь с таким email/телефоном/паспортными данными уже существует".<br>
            2с. Сценарий завершён неуспешно: Редактирование данных пользователя не выполнено.
        </td>
    </tr>
    <tr>
        <th>Негативный сценарий</th>
        <td>
            (Негативный: Ошибка сети при отправке формы)<br>
            1а. Система не получила данные формы из-за сбоя сети.<br>
            1b. Система выводит пользователю/менеджеру сообщение "Ошибка редактирования. Повторите попытку позже."<br>
            1с. Сценарий прерван: пользователь/менеджер не отредактировал данные.<br>
            (Другие возможные ошибки: потеря соединения с базой данных, ошибка в процессе валидации данных и т.д.)
        </td>
    </tr>
    <tr>
        <th>Конечное состояние</th>
        <td>
            При успешном сценарии — данные пользователя обновлены в системе. Пользователь получает уведомление о успешных изменениях.<br>
            При альтернативном/негативном — данные пользователя не обновлены, система возвращается в исходное состояние (сессия может быть завершена).
        </td>
    </tr>
    <tr>
        <th>Критерии приемки</th>
        <td>
            Пользователь с уникальными email и номером телефона может успешно редактировать данные, и эти данные сохраняются в системе. Система проверяет валидность данных; при ошибках показываются понятные подсказки. Пользователь получает сообщение на email и/или номер телефона о успешном изменении данных.<br>
            Все изменения данных клиента фиксируются в журнале аудита.
        </td>
    </tr>
    <tr>
        <th>Связанные требования</th>
        <td>
            BR-2.1 "API должен позволять редактировать информацию о клиентах (например, обновление адреса, номера телефона или электронной почты)."<br>
            BR-2.2 "Все изменения должны логироваться с указанием времени и ответственного сотрудника."<br>
            BR-4.1 "Все данные должны передаваться по защищённому каналу (например, через HTTPS)."<br>
            BR-5.1 "Все изменения должны быть зафиксированы в журнале аудита."
        </td>
    </tr>
</table>


<table>
    <tr>
        <th>Название</th>
        <td><b>UC-3. Удаление данных клиента</b></td>
    </tr>
    <tr>
        <th>Описание</th>
        <td>Менеджер выбирает необходимые данные клиента в административной панели, система удаляет необходимые данные, все действия логируются для аудита.</td>
    </tr>
    <tr>
        <th>Акторы</th>
        <td>Ответственный менеджер, Система</td>
    </tr>
    <tr>
        <th>Начальное состояние</th>
        <td>Менеджер находится в административной панели. У него есть задача удалить данные клиента.</td>
    </tr>
    <tr>
        <th>Триггер</th>
        <td>Менеджер решил удалить данные клиента (нажал "Удалить").</td>
    </tr>
    <tr>
        <th>Успешный сценарий</th>
        <td>
            1. Менеджер в административной панели выбирает необходимые данные, которые нужно удалить (ФИО, email, мобильный телефон, паспортные данные).<br>
            2. Система просит подтверждения действия (попап с предупреждением о необратимости операции).<br>
            3. Менеджер подтверждает удаление.<br>
            4. Система удаляет все необходимые данные пользователя и обновляет базу данных.<br>
            5. Все действия по удалению данных должны быть зафиксированы в журнале аудита с указанием времени и ответственного сотрудника.<br>
            6. Система отображает сообщение о том, что данные успешно удалены.
        </td>
    </tr>
    <tr>
        <th>Альтернативный сценарий</th>
        <td>
            (Альт.1: Пользователь отменяет удаление)<br>
            2а. Менеджер решает не удалять данные.<br>
            2б. Система отменяет удаление данных и возвращает менеджера в административную панель.<br>
            2с. Сценарий завершается без изменений в базе данных.
        </td>
    </tr>
    <tr>
        <th>Негативный сценарий</th>
        <td>
            (Негативный: Ошибка при удалении данных)<br>
            1а. Система не может удалить данные из-за ошибки базы данных (например, нарушение целостности данных).<br>
            1б. Система выводит сообщение об ошибке "Ошибка при удалении данных. Повторите попытку позже."<br>
            1с. Система не удаляет данные, база остаётся без изменений.
        </td>
    </tr>
    <tr>
        <th>Конечное состояние</th>
        <td>
            При успешном сценарии — данные пользователя удалены, база данных обновлена, пользователь уведомлен о выполнении операции.<br>
            При альтернативном сценарии — данные не удалены, система возвращает пользователя в административную панель.<br>
            При негативном сценарии — данные не удалены, система сообщает об ошибке.
        </td>
    </tr>
    <tr>
        <th>Критерии приемки</th>
        <td>
            1. Успешное удаление данных при подтверждении действия менеджером.<br>
            2. Система должна показывать сообщение о подтверждении и успешном удалении.<br>
            3. В случае ошибки должна быть предоставлена информация о причине сбоя (например, ошибка базы данных).<br>
            4. Данные не должны быть восстановлены после удаления.<br>
            5. Все действия должны быть зафиксированы в журнале аудита.
        </td>
    </tr>
    <tr>
        <th>Связанные требования</th>
        <td>
            BR-3.1 "API должно поддерживать возможность удаления данных о клиентах, но с учетом политики безопасности (например, сохранение минимального набора данных для юридических целей)."<br>
            BR-3.2 "Все действия по удалению данных должны быть зафиксированы и доступны для аудита."<br>
            BR-4.1 "Все данные должны передаваться по защищённому каналу (например, через HTTPS)."<br>
            BR-5.1 "Все действия должны быть зафиксированы в журнале аудита."
        </td>
    </tr>
</table>
